Shopping List API – How to Run & Test

This document describes how to run and test the Node.js / Express API implemented for the Shopping List homework. The focus of the assignment is on:

  • definition of dtoIn and dtoOut for each end-point (uuCmd),
  • validation of input data,
  • basic authorization based on application profiles,
  • returning uuAppErrorMap in responses.

1. How to run the server

  1. Open a terminal in the folder that contains server.js.
  2. Install dependencies: 
    npm install
  3. Start the server: 
    node server.js

If everything is correct, you will see: 

Shopping List API is running at http://localhost:3001

2. Application profiles (roles)

For the purpose of this homework, profiles are simulated in middleware and attached to req.user.profiles. The following profiles are defined:

  • Authorities – application administrators
  • User – any logged-in user
  • ShoppingListOwner – owner of a particular list
  • ShoppingListMember – member of a particular list

In the current demo configuration, the mocked user has profiles: User and ShoppingListOwner.

3. Implemented end-points (uuCmds)

Endpoint Method Description
/shoppingList/create POST Create a new shopping list
/shoppingList/get GET Get a shopping list by id
/shoppingList/list GET List shopping lists (mocked data)
/shoppingList/update POST Update (rename) a shopping list
/shoppingList/delete POST Delete a shopping list
/shoppingList/addMember POST Add a member to a shopping list
/shoppingList/removeMember POST Remove a member from a shopping list
/shoppingList/leave POST Current user leaves a list

Every handler performs:

  • validation of dtoIn,
  • authorization check based on profiles (for example, only owner can delete),
  • return of dtoOut with a uuAppErrorMap object.

4. Example requests

4.1. Create a shopping list POST /shoppingList/create

dtoIn

{
  "name": "My Homework List"
}

curl example

curl -X POST http://localhost:3001/shoppingList/create \
  -H "Content-Type: application/json" \
  -d "{\"name\": \"My Homework List\"}"

4.2. Get a shopping list GET /shoppingList/get

dtoIn (query): ?id=list-1

curl "http://localhost:3001/shoppingList/get?id=list-1"

4.3. List shopping lists GET /shoppingList/list

Query parameters pageIndex and pageSize are optional.

curl "http://localhost:3001/shoppingList/list?pageIndex=0&pageSize=10"

4.4. Update list name POST /shoppingList/update

dtoIn

{
  "id": "list-1",
  "name": "Updated List Name"
}

4.5. Delete a shopping list POST /shoppingList/delete

dtoIn

{
  "id": "list-1"
}

4.6. Add a member POST /shoppingList/addMember

dtoIn

{
  "listId": "list-1",
  "uuIdentity": "uu5:8765-4321",
  "role": "member"
}

4.7. Leave a list POST /shoppingList/leave

dtoIn

{
  "listId": "list-1"
}

5. Error handling – uuAppErrorMap

All responses follow the uuApp approach and contain a uuAppErrorMap object. On successful requests it is usually empty: 

{
  "awid": "shoppingListApp",
  "id": "list-123456789",
  "name": "My Homework List",
  "state": "active",
  "ownerUuIdentity": "uu5:1234-5678",
  "members": [],
  "items": [],
  "uuAppErrorMap": {}
}

In case of validation or authorization errors, entries are stored in uuAppErrorMap, e.g.: 

{
  "uuAppErrorMap": {
    "shoppingList/create/invalidName": {
      "type": "error",
      "message": "name is required and must be a non-empty string.",
      "paramMap": {
        "name": ""
      }
    }
  }
}

This demonstrates the required structure of input / output data and error reporting for the homework task.
